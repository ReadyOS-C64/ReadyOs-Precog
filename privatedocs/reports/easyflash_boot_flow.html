ReadyOS EasyFlash Boot Flow

Current-source walkthrough of the EasyFlash cartridge boot path: what executes where, when cartridge banking happens, how REU preload is built, and why the user sees a long blue-screen pause before the launcher appears.

Overview

Boot Model
Cart stub -> RAM loader -> REU snapshots -> launcher
Cart Size
64 banks x 16 KiB
App Window
$1000-$C5FF
Shim
$C800-$C9FF resident RAM
This EasyFlash flavor is not a permanent execute-from-cartridge design. The cartridge is primarily a cold-boot source. After preload, the launcher and apps are restored from REU into RAM and run from RAM.
On screen, this SKU now identifies itself during boot as precog cartridge (beta). The release folder and artifact names still remain precog-easyflash.

Memory Regions Used During Boot

RAM Execution / Staging

  • $0800-$27FF: EasyFlash loader copied into RAM and executed there
  • $1000-$C5FF: full app / launcher runtime window
  • $C600-$C7FF: REU bookkeeping and boot state
  • $C7F0-$C7FB: overlay metadata staging block
  • $C800-$C9FF: historical shim resident in RAM
  • $CC00-$CC8F: copied app layout table
  • $CC90-...: copied overlay layout table

Hardware / Mapped Areas

  • $8000-$9FFF: current cart ROML half
  • $A000-$BFFF: current cart ROMH half
  • $DE00: EasyFlash bank register
  • $DE01, $DE02: cart control registers
  • $DF01-$DF08: REU DMA registers
  • $0400: expected visible text screen RAM
  • $D020, $D021: border / background color

The background intentionally stays blue. The border is the progress indicator. Long yellow or orange periods are normal and mean the machine is actively preloading, not frozen.

Boot Flow

Stage 0: Power-on cartridge stub

The first code is the ROMH stub. It selects cart bank 0, copies 8 KiB from cartridge $8000 to RAM $0800, then jumps to $0800.

Executing: cartridge ROM. Copy destination: RAM.

Stage 1: Real loader starts from RAM

The main EasyFlash loader runs from RAM at $0800. It initializes CPU state, normalizes the CPU port, sets cart control registers, and prepares for preload work.

Stage 2: Loader tail refresh

The loader pulls in more of its own image from cart bank 0, copying from cartridge $8800 to RAM $1000 for $1800 bytes. This finishes assembling the full RAM-resident boot loader.

Stage 3: Boot screen prep

The loader clears screen RAM at $0400, writes boot text, and updates colors. However, the VIC may not yet be pointed at that screen RAM, so messages may exist logically without being visible.

The user-visible progress signal at this point is mainly the border-color change, not the screen text.

Stage 3A: Early REU requirement probe

Before shim install or any preload work, the loader performs a one-byte REU stash/fetch probe using RAM scratch at $CA11-$CA12 and REU bank $FE. If the byte comes back correctly, boot continues normally.

If the probe fails, the border switches to red, the loader reinitializes the machine into a stable text state, shows REU NOT DETECTED, EASYFLASH REQUIRES REU, and PRESS ANY KEY TO RETURN TO BASIC, then waits in a KERNAL GETIN loop and jumps to BASIC cold start at $FCE2 after keypress.

Stage 4: Historical shim install

The canonical 512-byte shim image is copied into RAM $C800-$C9FF. This becomes the runtime jump-table contract used by the launcher and later app switching.

$C800  load from disk and run
$C803  fetch from REU and run
$C806  just run $1000
$C809  preload to REU and return
$C80C  return to launcher
$C80F  switch app

The border switches to the shim color for this phase.

Stage 5: REU state init

The loader initializes bookkeeping in RAM around $C600-$C7FF, writes the REU magic marker, sets shim default drive state at $C839, and seeds the shared status bytes.

Stage 6: Copy generated layout tables to RAM

The app table is copied to $CC00 and the overlay table to $CC90. The preload loops then walk those RAM copies instead of repeatedly reading static metadata from cartridge-visible space.

Stage 7: Preload launcher snapshot

The launcher payload is copied from cart banks 1-2 into RAM at $1000, then the full app window $1000-$C5FF is stashed into REU bank 0.

The key nuance is that the cart payload is much smaller than the REU snapshot. The system stores a full runtime RAM image shape, not just the raw on-cartridge bytes.

The border switches to yellow for cart-to-RAM copy, then orange for the REU stash.

Stage 8: Preload 16 application snapshots

For each app, the loader clears $1000-$C5FF, selects that app's cart bank, copies the app payload into RAM, then stashes the full $B600-byte window to the assigned REU bank. It also marks the app as present in the shim bitmap at $C836-$C838.

During this loop, the border repeatedly alternates between yellow for cart reads and orange for REU stash work.

Stage 9: Preload 8 ReadyShell overlays

Overlays are staged through RAM starting at $1000, then stashed as $3800-byte blocks into REU banks 64 and 65 at fixed offsets. They are cached for later ReadyShell runtime use, not executed during boot.

The same yellow-then-orange border pattern is used here too.

Stage 10: Overlay metadata write

The loader builds a small overlay metadata block at $C7F0 and stashes it to REU bank $48 at offset $80F0.

Stage 11: Verify label

The screen says VERIFYING PRELOADS, but the current source does not perform a large late validation pass here. The label exists; the time cost is dominated by the earlier preload loops.

Stage 12: KERNAL and VIC normalization

The loader re-runs core KERNAL init, forces expected VIC banking and text-mode state, restores keyboard/editor state, and clears pending interrupts. This is necessary because EasyFlash startup may leave the machine in a nonstandard visual and channel state.

Stage 13: Restore launcher and hand off

The loader fetches REU bank 0 back into $1000-$C5FF and finally jumps to $1000. That is the handoff from boot asm to the C launcher.

The border switches to orange for the final restore, then light green for the last handoff into launcher code.

After cold boot, the steady-state runtime model is REU snapshot switching, not sustained direct cartridge execution.

Where Cartridge Banking Happens

Cartridge banking is concentrated in the boot preload path. The loader selects the current EasyFlash bank by writing to $DE00, then copies from $8000-$9FFF and, if needed, from $A000-$BFFF.

enable cart mapping
write selected bank to $DE00
copy ROML half from $8000-$9FFF
copy ROMH half from $A000-$BFFF if payload still remains
increment bank
repeat until payload length is satisfied
Once the launcher is restored and running, cartridge banking is mostly out of the hot path. Later launches use REU fetches back into RAM and shim calls from $C800.

Why The Blue Screen Pause Is So Long

The machine is not frozen during the long pause. It is building the runtime snapshot set up front.

  • Preloading launcher snapshot
  • Preloading 16 app snapshots
  • Preloading 8 overlay snapshots
  • Repeatedly clearing large RAM windows
  • Copying payloads from cartridge into RAM using 6502 software loops
  • Stashing those windows to REU

Roughly:

  • about 531 KiB of actual payload content copied from cartridge
  • about 885 KiB of snapshot / staging data stashed to REU
  • plus substantial repeated RAM clearing

With the new stage colors, the more accurate user-facing interpretation is:

  • blue background plus yellow border: cart payloads are being read
  • blue background plus orange border: snapshots are being pushed into REU
  • brief green border: shim-related setup work
  • brief light green border: launcher handoff is about to happen
The user mostly sees a blank blue stage because the boot code writes screen text before VIC screen banking is guaranteed to point at $0400, then later KERNAL text init rapidly clears or overwrites what becomes visible.

Summary

1. ROMH stub starts from cartridge and copies the real loader to RAM
2. RAM loader installs the historical shim at $C800-$C9FF
3. RAM loader preloads launcher, apps, and overlays from EasyFlash
4. RAM loader stashes those runtime windows into REU snapshots
5. Machine state is normalized for normal text-mode execution
6. Launcher snapshot is restored from REU bank 0 into $1000-$C5FF
7. Control jumps to $1000 and the C launcher begins
The large cold-boot delay is the price paid to make later launches effectively instant.
