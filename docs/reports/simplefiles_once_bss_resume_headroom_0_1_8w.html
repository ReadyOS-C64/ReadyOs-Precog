Failure mode: commit e0255e5 added a Simple Files linker config that reserves stack space but starts BSS at __ONCE_RUN__. That makes BSS overwrite the cc65 ONCE startup block at runtime. On first disk load the app can still start, but the REU snapshot later contains clobbered startup bytes. On reentry the shim restores those bytes and jumps to $1000; cc65 startup fails before main() and the machine returns to BASIC.
Fix: start BSS at __ONCE_RUN__ + __ONCE_SIZE__. This keeps the ONCE code intact inside the REU snapshot and keeps the existing fast warm-resume path. The only memory tradeoff is the correct 38-byte ONCE reservation.
24-app hypothesis checked: the 15-to-24 app-bank change was plausible, but it was not the root cause. The current memory contract check confirms SHIM_REU_BANK_APP_COUNT = 24, first dynamic REU bank 24, and snapshot length $B600.

Simple Files uses a custom cc65 app config with __HIMEM__ = $C600 and __STACKSIZE__ = $0070. The effective stack floor is $C590, so BSS/headroom accounting below measures free space from one byte after BSS to $C58F.

Headroom interpretation: the apparent pre-fix headroom was inflated by 38 bytes because BSS was illegally placed on top of ONCE. The post-fix build consumes those 38 bytes honestly. This is not a regression in usable memory; it is the cost of making the existing cc65 startup bytes durable across ReadyOS REU save/restore.

Resume Path Evidence

Failure before the fix

  • First disk launch entered Simple Files normally because ONCE bytes were present immediately after load.
  • Runtime BSS then reused $86FB-$8720, overwriting the ONCE startup block.
  • Returning to launcher saved the active app window $1000-$C5FF to the app REU bank, including the corrupted ONCE range.
  • Warm reentry restored the corrupted snapshot and jumped to $1000.
  • Temporary instrumentation showed main() was not reached after reentry, isolating the fault to cc65 startup before app code.

Proof after the fix

  • Built through bash ./run.sh --profile precog-dual-d71 readyshell-mon, producing the 0.1.8W D71 run.
  • Loaded Simple Files in the ReadyOS launcher, not by direct app launch.
  • Injected Ctrl+B through the C64 keyboard buffer; ReadyOS returned to the launcher.
  • Injected Return from the launcher; Simple Files resumed from the app snapshot.
  • Sampled again after 23 seconds; screen remained in Simple Files, not BASIC.
Acceptance result: the fix honors the quick app changing requirement. It does not force a reload on valid resume; it repairs the snapshot so the existing REU restore path can safely reenter cc65 startup.
