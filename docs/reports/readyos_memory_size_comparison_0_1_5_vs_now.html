Current 0.1.8 runtime reality: the active app window is $1000-$C5FF, shared metadata lives in $C600-$C7FF, and the resident shim lives in $C800-$C9FF.
Bank 0 correction: in current runtime terms, REU bank 0 is launcher-owned. It is not a generic “system metadata bank”; shared allocator and clipboard metadata are in C64 RAM.
Comparison scope correction: the historical 0.1.5 table only covers the apps that existed in that baseline snapshot. Current-only apps such as quicknotes, simplefiles, simplecells, and deminer are broken out separately below so the old/new totals stay honest.

Current 0.1.8 Memory Contract Notes

  • App runtime window: 0x1000-0xC5FF (46592 bytes).
  • REU metadata/system RAM: 0xC600-0xC7FF.
  • Shared hotkey bindings: 0xC7E0-0xC7E8.
  • Shim resident window: 0xC800-0xC9FF.
  • Bank 0 split: snapshot 0x0000-0xB5FF, resume tail 0xB600-0xFFFF.
  • Clipboard metadata lives in RAM; active clipboard payloads allocate one dynamic REU bank per item rather than using one packed shared bank.
  • Bitmap-tracked app-slot namespace: banks 0-23.
  • Dynamic allocator start: bank 24.
  • ReadyShell shared cache bank: 0x40 (core overlays 1-2).
  • ReadyShell REU debug bank: 0x43.
  • ReadyShell scratch/registry/value bank: 0x48.
  • Shipped media is profile-driven: cfg/profiles/*.ini generates the selected profile’s apps.cfg on drive 8.

These runtime facts supersede older low-bank, “system/launcher”, and “single shared clipboard bank” wording in earlier private reports.

Current Contract Checks

Current docs now reflect the conditional ReadyShell overlay contract: release overlay size $3800, debug overlay size $3B00.
Current ReadyShell loading model: overlays 1-2 share REU cache bank 0x40; command-only overlays are disk-loaded by default; the command registry and value arena live in bank 0x48.
  • launcher catalog is generated from the selected cfg/profiles/*.ini catalog to apps.cfg on drive 8
  • launcher catalog is read from apps.cfg by launcher only
  • launcher usually restores cached catalog state from bank 0 resume tail rather than rereading disk on app return
  • loaded-state tracking uses shim bitmap bytes at $C836-$C838
