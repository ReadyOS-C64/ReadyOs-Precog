ReadyOS SHIM Architecture: 0.2 Runtime Reality

Implementation-backed private report for the active launcher + shim + REU model. This canonical report is intentionally fuller than the short summary docs: it keeps the resident-byte map, the annotated shim routine walkthrough, the profile-driven catalog/release context, and the ABI-vs-implementation nuances that are easy to lose in condensed summaries. This report preserves the previous local snapshot values and adds the current map-backed values from the same public 0.2 release line after the ReadyShell overlay normalization. Runtime debug suffix letters are intentionally not used as documentation versions.

Why the layout matters

  • Everything in $1000-$C5FF is replaceable.
  • The shim survives all loads because it lives above the snapshot window.
  • The hardware stack survives preload and enables the shim’s RTS-based return path.
  • Metadata in $C600-$C9FF stays visible without fetching a management bank from REU.

Clipboard storage reality

  • Clipboard metadata in RAM: $C702 count, $C710 table.
  • Per-item table layout: each item stores REU bank id, type, and size.
  • Allocator behavior: clip_copy() calls reu_alloc_bank(REU_CLIPBOARD), so payload banks come from the dynamic pool starting at 24.
  • Legacy symbolic contract: some older headers/spec fields still name bank 1 as a clipboard bank, but active payload storage is not single-bank packed there.
Truthful current wording: clipboard metadata is shared in C64 RAM, and clipboard payloads still use one dynamic REU bank per item. The single-bank packed clipboard design has not been implemented yet.

`apps.cfg` and Launcher Lifecycle

Catalog reality

  • Human-edited source: selected cfg/profiles/<profile>.ini catalog
  • Generator/orchestrator: build_support/readyos_profiles.py
  • Disk payload: apps.cfg on drive 8
  • Runtime consumer: launcher only
  • Read mode: apps.cfg,s,r
  • Parsed fields: drive, PRG token, display label, optional hotkey slot, description
  • Assigned by launcher, not disk: REU bank identity

Reload behavior

Cold / invalid-resume path
launcher starts → tries bank 0 resume payload → failure → parse apps.cfg from disk → validate → save fresh LauncherResumeV1
Normal return path
app returns via shim → bank 0 launcher image restored → launcher restarts → launcher restores cached state from bank 0 tail → apps.cfg usually not reread
Boot never parses apps.cfg. Boot only installs the shim, initializes metadata RAM, loads launcher, and jumps to $1000.
Current release matrix: there are seven active profiles. precog-dual-d71 is the default full-content verification target and the only one that uses PREBOOT -> SETD71 -> BOOT. The other six current profiles use direct PREBOOT -> BOOT.

Linker-config nuance

  • Normal apps: cfg/ready_app.cfg puts BSS directly inside the $1000-$C5FF app window.
  • Overlay app: readyshell uses cfg/ready_app_overlay.cfg, not the normal app linker contract.
  • Historical resident ReadyShell image: the previous local snapshot ended at $9B74, with a much smaller resident image. Those values are retained only for historical comparison.
  • Current resident ReadyShell image: the current 0.2 map has resident BSS $8878-$8A6E (503 bytes), resident heap $8A70-$8DFD (910 bytes), and overlay load-address bytes at $8DFE-$8DFF.
  • True current ReadyShell headroom: overlay payload fit is tighter than the resident writable gap. Current overlay1 is $8E00-$C0CC and leaves 1331 bytes; overlay2 leaves 288 bytes; overlay3 leaves 6593 bytes; overlay4 leaves 3182 bytes; overlay5 leaves 5366 bytes.
  • Overlay guardrail: release overlay size is $3800 and debug overlay size is $3B00. The old explicit-C-stack-reserve wording is retained as historical shorthand, but current safety depends on ONCE/BSS ending below __OVERLAY_LOADADDR__, overlays loading at $8E00, and ReadyShell stack relocation being handled separately.

Transition Flows

Preload sandwich

1. Launcher writes target bank, filename, and drive into shim bytes.
2. Launcher calls JSR $C809.
3. Shim stashes launcher image to bank 0.
4. Shim KERNAL-loads target app into $1000.
5. Shim saves real load end to $C830-$C831.
6. Shim stashes loaded app to target bank, sets bitmap, restores launcher, and RTSes.

Return to launcher

1. App jumps to $C80C through TUI helper.
2. Shim stashes current app to its bank.
3. Shim updates bitmap and records last_saved.
4. Shim fetches launcher bank 0.
5. Shim sets current_bank = 0 and jumps to $1000.
6. Launcher restarts and usually restores cached menu/catalog state from the bank 0 tail.

Launch from REU

1. Launcher saves LauncherResumeV1.
2. Launcher stashes itself to bank 0.
3. Launcher writes target scratch bytes and sets current_bank.
4. Launcher jumps to $C803.

Launch from disk

1. Launcher saves LauncherResumeV1.
2. Launcher stashes itself to bank 0.
3. Launcher writes filename and patched drive bytes.
4. Launcher sets current_bank.
5. Launcher jumps to $C800.
