ReadyShell Architecture

End-to-end architecture for the current ReadyShell implementation in ReadyOS. This document is implementation-accurate for the local build and complements the generated overlay inventory report.

System Shape

ReadyShell is an overlayed C64 app with resident control code, a parse overlay, an execution-core overlay, and command overlays. The resident shell owns orchestration; parsing and most execution live in the shared overlay window.

Resident growth reduces heap directly. Overlay growth does not reduce heap, but every overlay must still fit the 0x3800 window.

Command Families

ReadyShell commands split into exec-core commands and external commands.

  • Exec-core commands in OVERLAY2: PRT, MORE, TOP, SEL, GEN, TAP
  • External commands in command overlays: DRVI, LST, LDV, STV, DEL, REN, PUT, ADD, CAT, COPY

External commands are driven through one generic resident execution path plus the REU-backed command registry.

Pipeline And Parse Flow

source line
   |
   v
OVERLAY1: lexer + parser
   |
   v
resident regains control
   |
   v
OVERLAY2: execute pipeline stage-by-stage
   |
   +--> expr stage
   +--> filter stage
   +--> foreach stage
   +--> exec-core command
   `--> external command

The overlay command ABI defines BEGIN, ITEM, RUN, PROCESS, and END. The current shipped commands use RUN for most command overlays and BEGIN + ITEM for CAT.

External Command Registry

RegionRangePurpose
Registry header$488010-$488017magic, version, counts
Command descriptors$488020-$48807Fexternal command metadata
Overlay state table$488080-$4880D9overlay load/cache state

The registry is stored in REU metadata rather than resident heap/BSS, so new external commands do not require resident per-command wrapper growth.

External Command Call Flow

OVERLAY2 decides command is external
    |
    v
resident vm_cmd_external()
    |
    v
lookup descriptor in REU registry
    |
    v
prepare external overlay from overlay-state record
    |
    v
load overlay from disk into overlay window
    |
    v
call one overlay dispatcher with handler id
    |
    v
command overlay runs command body
    |
    v
resident restores OVERLAY2 from REU

Repeating LST repeats the disk load. A file command such as COPY "8:notes","9:notes" follows the same pattern: load rscat, run the handler, then restore rsvm from REU. The registry stores metadata about external overlays; it does not make them REU-cached automatically.

These are full-window snapshots, not just file payloads, so writable overlay data survives switching.

Responsibilities

  • Resident code owns the shell loop, overlay boot/load/restore logic, generic external command dispatch, screen/platform glue, and REU coordination.
  • Overlay 1 owns lexing, parsing, parse support, and cleanup.
  • Overlay 2 owns runtime values, variables, formatting, pipes, command lookup, capability classification, and shared execution helpers.
  • Command overlays own command-specific disk logic, serialization logic, and overlay-local dispatchers.

Current Constraints

  • Resident heap below the overlay load address is only 910 bytes.
  • OVERLAY2 is the tightest overlay and remains the next likely growth bottleneck.
  • External command overlays still disk-load on every call unless a later change adds REU caching for them.
  • The registry removes resident per-command glue, but command-name and capability growth in OVERLAY2 still matters.
